# REDUND-03: Strip Analyst Diagnostic Fields from Page Persistence

**Priority**: Quick Win
**Effort**: S
**Dependencies**: None
**Category**: Storage redundancy

## Summary

Multiple analyst result sub-fields are generated by the LLM for chain-of-thought reasoning quality but never read in production. Keep them in the LLM output schema (the LLM needs to produce them for good analysis) but strip them before persisting the page.

## Problem

The following `analystResult` fields are never read after page assembly:

**From StructureEvaluatorResult:**
- `beatResolution` — text summary of how beat resolved
- `objectiveAnchors` — evidence items for beat conclusion
- `anchorEvidence` — supporting evidence text
- `completionGateSatisfied` — whether gate was met
- `completionGateFailureReason` — why gate wasn't met
- `alignedBeatId` — which beat scene aligns to
- `beatAlignmentConfidence` — confidence level
- `beatAlignmentReason` — reason text

**From NpcIntelligenceResult:**
- `npcCoherenceAdherent` — whether NPCs were coherent
- `npcCoherenceIssues` — coherence issue text
- `relationshipShiftsDetected` — detected shifts (relationship updates come from dedicated page fields)

These fields serve as LLM "thinking" — they help the analyst reason better. They should remain in the JSON schema sent to the LLM. But they don't need to be stored on disk.

## Proposed Fix

1. Keep all fields in the LLM output schemas (`src/llm/schemas/`)
2. Keep all fields in the result types (they're used transiently during page assembly)
3. In `page-serializer.ts`, strip these fields from `analystResult` before writing
4. On deserialization, these fields will naturally be `undefined` (backward compatible)

## Files to Touch

- `src/persistence/page-serializer.ts` — strip diagnostic fields from `analystResult` before serialization
- `test/` — update any tests that assert on these fields in deserialized pages

## Out of Scope

- Changing the LLM output schema (must keep for reasoning quality)
- Changing how the engine uses these fields transiently during page assembly
- Modifying the analyst prompt

## Acceptance Criteria

- [ ] `npm run typecheck` passes
- [ ] Serialized page JSON no longer contains stripped diagnostic fields
- [ ] LLM schemas unchanged (analyst still produces these fields)
- [ ] Old page files with these fields still deserialize without error
- [ ] All existing tests pass

## Outcome

**Decision date**: 2026-03-07
**Disposition**: NOT IMPLEMENTED — Instead of stripping these fields, they were surfaced in the Story Insights modal as part of the "Director's Commentary" redesign. The modal now has 4 tabs (Structure, Narrative, NPCs, Payoffs) that display all analyst data including beatResolution, objectiveAnchors, anchorEvidence, completionGate fields, npcCoherence fields, relationshipShifts, dramaticIronyOpportunities, thematicCharge, narrativeFocus, and deviation data. Keeping all fields in persistence supports this enriched UI.
