## 2. Property Lifecycles

### 2.1 StoryKernel: `valueSpectrum` + `moralArgument`

**Origin**: Generated by the kernel ideator LLM call.

```
kernel-ideator-prompt.ts  →  kernel-ideator-schema.ts  →  StoryKernel model
```

The ideator prompt instructs the LLM to produce a `valueSpectrum` object with four degrading value levels (McKee's value charge: positive → contrary → contradictory → negation of negation) and a `moralArgument` (the story's dramatic conclusion thesis).

**Validation**: `isStoryKernel()` in `story-kernel.ts` requires both fields. `isValueSpectrum()` validates all four sub-fields are non-empty strings.

**Downstream consumption** (broadest propagation of any new property):

| Stage | File | How Injected |
|-------|------|--------------|
| Kernel evaluation | `kernel-evaluator-prompt.ts` | Evaluated kernels carry these fields for ranking |
| Kernel evolution | `kernel-evolver-prompt.ts` | Parent kernels carry these fields; 4 new mutation strategies reference them |
| Concept ideation | `concept-ideator-prompt.ts` | All 4 spectrum values + moralArgument injected as grounding |
| Concept seeding | `concept-seeder-prompt.ts` | Same injection |
| Concept architecture | `concept-architect-prompt.ts` | Same injection |
| Concept engineering | `concept-engineer-prompt.ts` | Same injection |
| Concept evolution | `concept-evolver-prompt.ts` | Same injection |
| Concept evolver seeding | `concept-evolver-seeder-prompt.ts` | Same injection |
| Spine generation | `spine-prompt.ts` | Injected in kernel grounding section |
| Entity decomposition | `entity-decomposer-prompt.ts` | All 4 spectrum values + moralArgument |
| Structure generation | `structure-prompt.ts` | Full VALUE SPECTRUM (McKee) block with per-act calibration instruction |
| Opening page planner | `sections/planner/opening-context.ts` | `buildOpeningValueSpectrumSection()` |
| Continuation planner | `sections/planner/continuation-context.ts` | `buildValueSpectrumGuidanceSection()` |
| UI: kernel cards | `04e-kernel-renderer.js` | Renders all 4 spectrum values + moralArgument |
| UI: new-story page | `09-controllers.js` | `renderStoryKernelDisplay()` populates kernel panel |

**Not injected into**: lorekeeper, writer, analyst prompts. This is by design — the lorekeeper curates from existing story context (spine + decomposed data already encode thematic direction), and the writer/analyst operate at scene level where the spine's `centralDramaticQuestion` provides sufficient thematic grounding.

### 2.2 KernelDimensionScores: `ironicPotential` + `viscerality`

**Origin**: Generated by the kernel evaluator LLM call.

```
kernel-evaluator-prompt.ts  →  kernel-evaluator-schema.ts  →  KernelDimensionScores
```

**Lifecycle**: These are **selection-gate properties**. They exist solely to score and rank kernels during the kernel pipeline (ideation → evaluation → evolution). Once a kernel passes thresholds and is selected, these scores are not propagated downstream — the kernel's `valueSpectrum` and `moralArgument` carry the thematic content forward instead.

| Stage | Role |
|-------|------|
| Kernel evaluation | Scored 0-5 with evidence arrays |
| `computeKernelOverallScore()` | Weighted into overall score (15% + 10%) |
| `passesKernelThresholds()` | Must be ≥ 3 each |
| Kernel evolution | Parent scores visible to evolver for mutation targeting |
| UI: kernel cards | Displayed in score grid |

### 2.3 ConceptSpec: `protagonistLie`, `protagonistTruth`, `protagonistGhost`, `wantNeedCollisionSketch`

**Origin**: Generated across the 3-stage concept pipeline.

```
Stage 1 (Seeder/Ideator): generates seed fields (no Weiland fields yet)
Stage 2 (Architect): generates character/world fields (no Weiland fields yet)
Stage 3 (Engineer): generates protagonistLie, protagonistTruth, protagonistGhost, wantNeedCollisionSketch
```

In the monolithic ideator path, all fields are generated in one call. In the 3-stage pipeline, the engineer is responsible for generating these four fields, with constraints:
- `protagonistLie` must be the false belief that the `deadlineMechanism` exploits
- `protagonistTruth` must dissolve the Lie and resolve the `moralArgument`
- `protagonistGhost` must make the `pressureSource` personally devastating

**Validation**: `isConceptSpec()` requires all four as non-empty strings. `SavedConcept` persistence validates via the same guard.

## 3. Architecture Analysis: Why Weiland Fields Stop at Spine/Entity Decomposition

The ConceptSpec Weiland fields (`protagonistLie`, `protagonistTruth`, `protagonistGhost`, `wantNeedCollisionSketch`) are **not** directly injected into the per-page generation pipeline (structure, planner, writer, lorekeeper, analyst). This is intentional and architecturally sound:

1. **Transformation, not passthrough**: The concept is an intermediate design artifact. Its Weiland fields are *consumed and transformed* by two downstream generators:
   - **Spine generator**: Transforms `wantNeedCollisionSketch` → `wantNeedCollisionPoint` (concrete dramatic moment) and `protagonistGhost` → `protagonistDeepestFear` (operationalized fear). These spine fields then propagate to every prompt via `buildSpineSection()`.
   - **Entity decomposer**: Absorbs `protagonistLie`/`Truth`/`Ghost` into structured character profiles (`decomposedCharacters`), which are passed to every downstream prompt.

2. **Information is preserved, not lost**: The Lie/Truth/Ghost triad reaches the writer and analyst through two channels:
   - The **spine** carries the collision point and deepest fear (derived from ghost + lie)
   - The **decomposed characters** carry the protagonist's wound, flaw, and internal conflict (derived from all four Weiland fields)

3. **Avoiding redundant context**: Injecting raw concept fields alongside their transformed spine/character versions would create conflicting guidance for the LLM, increasing prompt token cost without improving output quality.

### Similarly for kernel `valueSpectrum`/`moralArgument`

These ARE directly injected into the planner (both opening and continuation) via dedicated section builders, and into the structure prompt. They are NOT in lorekeeper/writer/analyst because:
- The **spine's `centralDramaticQuestion`** (derived from the kernel) provides scene-level thematic grounding
- The **structure's act/beat objectives** (generated from kernel + spine) encode the value progression
- Adding raw kernel abstraction to scene-level prompts would compete with the more concrete spine guidance

## 4. Potential Gaps Identified

### 4.1 Gap: `ConceptEngineFields` Pick Type — Verified Complete

The `ConceptEngineFields` type (used by the 3-stage concept pipeline's engineer stage) correctly includes all four Weiland fields in its `Pick`:

```typescript
export type ConceptEngineFields = Pick<
  ConceptSpec,
  | 'pressureSource' | 'stakesPersonal' | 'stakesSystemic' | 'deadlineMechanism'
  | 'ironicTwist' | 'incitingDisruption' | 'escapeValve' | 'elevatorParagraph'
  | 'protagonistLie' | 'protagonistTruth' | 'protagonistGhost' | 'wantNeedCollisionSketch'
>;
```

No gap here.

### 4.2 Gap: SavedConcept Upcasting for Legacy Payloads

`SavedConcept` persistence uses a `parseEntity` hook to upcast legacy payloads missing newer fields. This was already in place for `incitingDisruption` and `escapeValve`. The new Weiland fields (`protagonistLie`, `protagonistTruth`, `protagonistGhost`, `wantNeedCollisionSketch`) and new scoring dimensions (`ironicPremise`, `sceneGenerativePower`) will cause `isConceptSpec()` and `isConceptDimensionScores()` to **reject** any legacy saved concepts that lack these fields.

**Risk**: If a user has saved concepts from before this overhaul, loading them will fail validation.

**Mitigation needed**: The `parseEntity` hook in the concept repository should upcast legacy payloads by providing default values for missing fields. This needs verification.

### 4.3 Gap: Story Serializer Legacy Compatibility for Spine Fields

The story serializer correctly handles legacy spines with `?? ''` fallback for `wantNeedCollisionPoint` and `protagonistDeepestFear`. Existing stories with old-format spines will deserialize with empty strings. **No action needed** — empty strings are valid and will simply not display in the UI (the renderer checks `if (option.wantNeedCollisionPoint)`).

### 4.4 No Gap: Kernel Persistence

Kernels use a separate repository with `isStoryKernel()` validation. Legacy kernels missing `valueSpectrum`/`moralArgument` will fail validation. However, the kernel repository's `parseEntity` hook should handle upcasting. This needs the same verification as 4.2.

## 5. Confirmed Bugs: Legacy Persistence Will Break

Investigation confirmed that legacy saved kernels and concepts **will fail to load** after this overhaul. This affects any user who has saved kernels or concepts from before the overhaul.

### 5.1 Bug: Legacy Saved Concepts Cannot Load

**File**: `src/persistence/concept-payload-parser.ts`

The `parseSavedConcept()` function does zero upcasting — it calls `isSavedConcept()` and throws on failure. Since `isSavedConcept()` delegates to `isConceptSpec()` (which now requires `protagonistLie`, `protagonistTruth`, `protagonistGhost`, `wantNeedCollisionSketch`) and `isConceptDimensionScores()` (which now requires `ironicPremise`, `sceneGenerativePower`), any legacy concept missing these fields will throw `Error: Invalid SavedConcept payload`.

**Impact**: `listConcepts()`, `loadConcept()`, and `updateConcept()` will all fail for pre-overhaul concepts.

**Fix needed in `concept-payload-parser.ts`**: Before calling `isSavedConcept()`, upcast the raw payload by injecting default values for missing fields:

```typescript
// ConceptSpec fields — need non-empty strings to pass isConceptSpec()
concept.protagonistLie ??= '[Legacy — not yet generated]';
concept.protagonistTruth ??= '[Legacy — not yet generated]';
concept.protagonistGhost ??= '[Legacy — not yet generated]';
concept.wantNeedCollisionSketch ??= '[Legacy — not yet generated]';

// ConceptDimensionScores — need valid numbers
scores.ironicPremise ??= 0;
scores.sceneGenerativePower ??= 0;

// ConceptScoreEvidence — need non-empty string arrays
evidence.ironicPremise ??= ['Legacy concept — not evaluated'];
evidence.sceneGenerativePower ??= ['Legacy concept — not evaluated'];
```

### 5.2 Bug: Legacy Saved Kernels Cannot Load

**File**: `src/persistence/kernel-repository.ts`

The kernel repository has **no `parseEntity` hook at all** — it falls through to the default `isEntity` asserter, which calls `isSavedKernel()`. This delegates to `isStoryKernel()` (which now requires `valueSpectrum` and `moralArgument`). Legacy kernels missing these fields will throw `Error: Invalid SavedKernel payload`.

**Fix needed**: Add a `parseEntity` hook to the kernel repository (similar to `concept-payload-parser.ts`) that upcasts legacy payloads:

```typescript
// StoryKernel fields
kernel.valueSpectrum ??= {
  positive: '[Legacy — not yet generated]',
  contrary: '[Legacy — not yet generated]',
  contradictory: '[Legacy — not yet generated]',
  negationOfNegation: '[Legacy — not yet generated]',
};
kernel.moralArgument ??= '[Legacy — not yet generated]';
```

### 5.3 Bug: `saved-kernel.ts` Validator Missing New Score Dimensions

**File**: `src/models/saved-kernel.ts`, lines 52-63

The `isKernelDimensionScores()` validator in `saved-kernel.ts` only checks the original 5 dimensions:

```typescript
// CURRENT (missing ironicPotential and viscerality):
function isKernelDimensionScores(value: unknown): boolean {
  return (
    isScore(value['dramaticClarity']) &&
    isScore(value['thematicUniversality']) &&
    isScore(value['generativePotential']) &&
    isScore(value['conflictTension']) &&
    isScore(value['emotionalDepth'])
  );
}
```

This is a **validator inconsistency** — the model's `KernelDimensionScores` interface requires 7 dimensions, but the saved-kernel validator only checks 5. While this doesn't cause loading failures (it's permissive rather than restrictive), it means invalid data could pass validation. A newly-saved kernel with all 7 dimensions will load fine, but if `ironicPotential` or `viscerality` were somehow corrupted or missing, the validator would not catch it.

**Fix needed**: Add `ironicPotential` and `viscerality` checks to `isKernelDimensionScores()` in `saved-kernel.ts`. Then also add upcasting defaults in the kernel parser for legacy payloads:

```typescript
scores.ironicPotential ??= 0;
scores.viscerality ??= 0;
```

### 5.4 Resolution

No legacy compatibility / upcasting is desired. The `kernels/` and `concepts/` directories contain no pre-existing JSON files, so there is no data to migrate.

The only actionable fix was the **validator inconsistency** in `saved-kernel.ts`:

| File | Fix | Status |
|------|-----|--------|
| `src/models/saved-kernel.ts` | Added `ironicPotential` + `viscerality` checks to `isKernelDimensionScores()` | **DONE** |
| `src/persistence/concept-payload-parser.ts` | No change needed — no legacy data exists | N/A |
| `src/persistence/kernel-repository.ts` | No change needed — no legacy data exists | N/A |
